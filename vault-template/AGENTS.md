# AGENTS.md — DuckSidian Wiki Schema

Этот файл — **главный конфиг агента**. Прочитай его целиком перед каждой
операцией (ingest / query / lint / summary). Здесь описано: домен, структура
vault'а, правила, форматы страниц, conventions.

## Мультичат

DuckSidian может отслеживать несколько чатов одновременно. Каждый чат
изолирован в собственной поддиректории vault'а:

```
vault/
  main/        ← чат "main"  (raw/, wiki/, AGENTS.md, ...)
  sales/       ← чат "sales"
  ...
```

Агент всегда работает в контексте одного чата — путь передаётся в user_prompt.
Не смешивай данные разных чатов: wikilinks внутри одного чата — относительны
от его корневого каталога.

## Domain

Этот wiki — корпоративная память **рабочего чата команды в Telegram**.
Источник истины — переписка в супергруппе. Цель — превратить хаотичный поток
сообщений в навигируемую базу знаний: кто что решил, какие сущности упоминаются
(проекты, люди, репозитории, клиенты, технологии), какие концепции
обсуждаются, как они связаны.

## Three layers (по Карпатому)

1. **`raw/`** — immutable source of truth. Агент **только читает** отсюда.
   - `raw/daily/YYYY-MM-DD.md` — дневной батч сообщений (создаётся ботом).
   - `raw/notes/<ts>-<slug>.md` — ручные заметки через `/note`.
   - `raw/assets/` — медиа из чата (фото, видео, документы, аудио).
     Структура: `raw/assets/<chat_name>/<YYYY-MM-DD>/<filename>`.
     Агент **только читает** ссылки; реальная запись файлов выполняется ботом.
2. **`wiki/`** — markdown-страницы, **которые пишет агент**. Здесь живёт всё
   синтезированное знание.
   - `wiki/projects/<Slug>.md` — **проекты**, над которыми работает команда
     (изделия, заказы, R&D-направления). Это первоклассный тип: при любом
     упоминании проекта в чате агент обязан создать/обновить эту страницу.
   - `wiki/entities/<Slug>.md` — люди, репозитории, продукты, клиенты,
     поставщики (всё, что **не** проект).
   - `wiki/concepts/<Slug>.md` — технические идеи, паттерны, термины, решения.
   - `wiki/sources/<Slug>.md` — внешние источники (статьи, PR, тикеты), на
     которые ссылались в чате.
   - `wiki/daily/YYYY-MM-DD.md` — дайджест дня (executive summary батча).
   - `wiki/queries/<ts>-<slug>.md` — сохранённые ответы на `/ask`, если
     полезные.
3. **Schema** — этот файл + `index.md` + `log.md`.

## Conventions

### Медиафайлы
Когда в батче есть фото/видео/документ/аудио, raw/daily/*.md содержит строку:
```
> 📎 *photo*: `[[raw/assets/main/2026-04-28/photo_1234.jpg]]`
```
- В wiki/ упоминай как attachment: `![[raw/assets/.../file.jpg]]` для изображений.
- Для документов — ссылка: `[[raw/assets/.../document.pdf]]`.
- Содержимое медиа не анализируй, если не можешь прочитать файл.

### Wikilinks
- Всегда используй `[[wiki/entities/Имя]]` или `[[wiki/concepts/Тема]]` для
  ссылок на страницы. Не используй внешние URL для внутренних связей.
- Пути — относительные от корня vault'а.

### Citations
Каждое утверждение в `wiki/` **должно** ссылаться на источник в `raw/`:

```
Команда решила перейти на pgvector ([[raw/daily/2026-04-15#msg-1234]]).
```

`#msg-<id>` — telegram message_id из дневного батча. Бот размечает каждое
сообщение в `raw/daily/*.md` якорем `^msg-<id>`.

### YAML frontmatter
У каждой страницы в `wiki/` обязателен frontmatter:

```yaml
---
type: entity | concept | source | daily | query
created: 2026-04-28
updated: 2026-04-28
sources: 3   # сколько уникальных raw/* подтверждают эту страницу
tags: [project, backend]
aliases: ["pgvector", "pg_vector"]
---
```

### Стиль
- Заголовок страницы = `# Имя` (совпадает с filename slug).
- Первый абзац — одна строка-определение.
- Дальше секции `## Что это`, `## Контекст`, `## Связи` (список wikilinks),
  `## Открытые вопросы`, `## История` (chronological bullets с цитатами).
- Не выдумывай факты. Если в `raw/` нет подтверждения — не пиши.
- При противоречии между источниками — пометь `> ⚠️ Противоречие: …` и
  оставь оба варианта с цитатами.

### Проекты (`wiki/projects/`)

Проект — это любое именованное **направление работы команды**: изделие,
заказ, R&D-тема, хардварный узел, версия продукта (например, `SIRIUSROVER`,
`UNEX-UGV`, `Cycloidal-wheel`). Признаки в чате:
- повторяющееся имя (с версиями: `v10`, `v14`, …);
- обсуждение требований/проблем/решений вокруг этого имени;
- прикреплённые артефакты (CAD-файлы, чертежи, фото).

Правила:
- Шаблон страницы — в `wiki/projects/_template.md` (этот файл не считается
  проектом, lint его игнорирует).
- Frontmatter: `type: project`, `status: active|paused|done|archive`,
  `owners: [...]`, `aliases: [...]`.
- Обязательные секции: `## Цели и контекст`, `## Текущий статус`,
  `## Открытые задачи`, `## Решения`, `## Проблемы`, `## Артефакты`,
  `## Связи`, `## История`. Не удаляй секции — оставляй пустыми, если нет
  данных.
- При каждом ingest, где проект упомянут:
  1. Если страницы нет — создать из шаблона.
  2. Дописать в `## История` буллет с датой и `[→ tg](...)`.
  3. Обновить `## Текущий статус` (одна свежая фраза).
  4. Если в обсуждении явная задача/проблема/решение/артефакт — занести
     в соответствующую секцию.
  5. Поднять `sources` и `updated` во frontmatter.
- Если статус давно не двигался (нет упоминаний > 30 дней) — `status: paused`.
- Алиасы версий (`SIRIUSROVER v10`, `SIRIUSROVER v14`) ведут на одну
  страницу проекта `SIRIUSROVER`. Версии перечисляй внутри страницы.

## Operations

### Ingest (запускается ботом по cron или /ingest)

Вход: путь к `raw/daily/YYYY-MM-DD.md`. Также доступны медиафайлы в `raw/assets/`.

Шаги агента:
1. Прочитай `AGENTS.md` (этот файл), `index.md`, последние 30 строк `log.md`.
2. Прочитай дневной батч.
3. Извлеки: ключевые решения, упомянутые **проекты**, сущности, новые
   концепции, ссылки на внешние источники.
4. Для каждого упомянутого проекта — обнови/создай `wiki/projects/<Slug>.md`
   по правилам из раздела «Проекты».
5. Для каждой сущности/концепции:
   - если страница уже есть в `wiki/` — **обнови** её (новые цитаты,
     обновлённое определение, новые связи).
   - если нет — **создай** новую с frontmatter и базовой структурой.
6. Создай `wiki/daily/YYYY-MM-DD.md` — дайджест дня (3–10 буллетов с
   wikilinks; для активных проектов — отдельная подсекция).
7. Обнови `index.md`: добавь новые страницы в соответствующие секции
   (включая `## Projects`).
8. Допиши строку в `log.md`:
   `## [YYYY-MM-DD HH:MM] ingest | <N> сообщений → <K> страниц обновлено`
9. Если что-то неоднозначно (например, два похожих имени — это один проект
   или два?) — вызови tool `ask_user(question, options)`.
   Не пиши в wiki сомнительные факты.

### Query (`/ask <вопрос>`)
1. Прочитай `index.md`.
2. Найди релевантные страницы (по названиям и тегам в индексе).
3. Прочитай их, сформулируй ответ с inline wikilinks и цитатами.
4. Ответь в чат.

### Lint (`/lint`)
Проверь и сообщи:
- противоречия между страницами,
- orphan-страницы (нет inbound links),
- стэйл-страницы (`updated` старше 90 дней + sources не растут),
- упомянутые но не созданные сущности (broken `[[wikilinks]]`),
- пробелы в `index.md`.
- Допиши итог в `log.md`.

### Summary (`/summary day|week`)
Прочитай `wiki/daily/*` за период, выдай сводку **в чат** (без записи в
vault).

## Constraints

- **НИКОГДА** не модифицируй `raw/` (read-only с твоей стороны).
- **НИКОГДА** не пиши вне `/vault`. Доступные через tools пути всегда
  относительные от корня vault'а.
- Один ingest = один транзакционный набор изменений → один git-коммит.
- Если батч слишком большой — сначала составь промежуточный конспект, потом
  ingest по конспекту.
